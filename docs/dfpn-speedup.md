# df-pn 検証の高速化調査

## 現状の課題

パズル生成において、df-pn による検証（詰み判定 → 短手数不存在 → 唯一解チェック）がボトルネック。
特に 9手詰以上では候補生成は速い（逆算法で 1000局面/0.08秒）が、検証に時間がかかる。

### 生成実績

| 手数 | 問題数 | 50,000 attempts | 所要時間 |
|------|--------|-----------------|----------|
| 9手詰 | 16問 | +11問 | 約12時間 |
| 11手詰 | 0問 | +0問 | 約3時間 |

## 高速化テクニック一覧

### 1. n手詰めルーチン（推奨: 最初に実装）

**概要**: 1手詰・3手詰を専用ルーチンで即判定し、df-pn の再帰を回避する。

**原理**: df-pn は末端ノード (pn=1, dn=1) から証明数を積み上げるが、
浅い詰みは専用コードで O(合法手数) で判定できる。
df-pn のハッシュ参照・閾値更新のオーバーヘッドを完全にスキップできる。

**実装方針**:

#### 1手詰ルーチン

```
fn is_mate_in_1(state: &State) -> bool:
    攻め方の王手になる手を列挙
    各王手に対して:
        手を適用
        守り方に合法手がなければ → 詰み（true を返す）
        手を戻す
    false
```

- `generate_checks` + `has_any_legal_move` の組み合わせ
- ハッシュテーブル不要、再帰不要

#### 3手詰ルーチン

```
fn is_mate_in_3(state: &State) -> bool:
    攻め方の王手になる手を列挙
    各王手に対して:
        手を適用
        守り方の合法手を列挙
        全ての守り方の手に対して:
            手を適用
            is_mate_in_1 で判定
            手を戻す
        全て詰みなら → true
        手を戻す
    false
```

#### df-pn への組み込み

`mid_or` / `mid_and` の末端判定を変更:
- `depth == 1` の OR ノード: `is_mate_in_1` で即判定
- `depth <= 3` の OR ノード: `is_mate_in_3` で即判定
- 成功時は `store(key, 0, INF)`、失敗時は `store(key, INF, 0)` でTTに記録

**期待効果**: 中〜大。探索木の末端ノードの大半を高速処理できる。
特に長手数の探索では末端ノードが膨大なため、効果が大きい。

**実装難易度**: 低。既存の `generate_checks` と `has_any_legal_move` を再利用可能。

### 2. 証明駒・反証駒（効果: 非常に大きい / 実装: 複雑）

**概要**: 詰みを証明するのに必要な最小限の攻め方持ち駒を追跡する。

**原理**:
- 証明駒 = ある局面で詰みを示すために必要な攻め方の最小持ち駒セット
- 反証駒 = 不詰を示すために必要な攻め方の最小持ち駒セット
- 優等局面（持ち駒が多い局面）は自動的に詰みと判定 → 再探索不要

**計算方法**:
- OrNode（攻め方）: `proof_pieces = Before(child_proof_pieces)` — 子の証明駒から手の持ち駒増減を逆算
- AndNode（受け方・近接王手）: `proof_pieces = ∪ child_proof_pieces` — 全子ノードの和集合
- AndNode（受け方・遠隔王手）: `proof_pieces = (∪ child_proof_pieces) ∪ monopolized_pieces`
  - 独占駒種: 攻め方が全数を持っているため守り方が合駒に使えない駒種

**実装上の注意**:
- 持ち駒をビット表現すれば、優等判定が数命令で可能
- 遠隔王手（香・飛・角）時の合駒判定で独占駒種の特殊処理が必要
- 千日手（連続王手の千日手）は対象外とする
- 証明駒は一意ではない（複数の極小セットが存在しうる）

**期待効果**: 非常に大きい。同一盤面で持ち駒だけ異なる局面の再探索を回避できる。

**参考**: [証明駒／反証駒の活用方法](https://komorinfo.com/blog/proof-piece-and-disproof-piece/)

### 3. 閾値制御の最適化（効果: 中 / 実装: 低）

**現状**: 現在の実装は正しい閾値計算を行っている。

```rust
// OR ノード
c_th_pn = th_pn.min(second_pn + 1)
c_th_dn = th_dn - sum_dn + child_dn[best]

// AND ノード
c_th_dn = th_dn.min(second_dn + 1)
c_th_pn = th_pn - sum_pn + child_pn[best]
```

現状の実装は標準的な df-pn の閾値制御に準拠している。大きな改善余地は少ない。

### 4. 手の並び替えの改善（効果: 小〜中 / 実装: 低）

**現状**: 捕獲+成り > 捕獲 > 成り > 移動 > 打ち の順でソート済み。

**追加の改善案**:
- 玉に近い駒からの王手を優先
- 玉の退路を塞ぐ手を優先
- 前回の探索で有効だった手を記録（キラーヒューリスティック）

### 5. 探索の並列化（効果: 大 / 実装: 高）

df-pn の並列化は非自明だが、独立した候補局面の検証は並列化可能（現在実施済み）。
df-pn 内部の並列化は GHI 問題などとの兼ね合いで複雑。

## 推奨実装順序

1. **n手詰めルーチン**（1手・3手） — 実装が簡単で効果が見込める
2. **証明駒・反証駒** — 効果は最大だが実装が複雑
3. **手の並び替え改善** — 小さな改善を積み重ね
