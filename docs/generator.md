# 生成エンジンの実装解説

パズル生成エンジン (`tsume-gen`) は、詰将棋パズルの自動生成と検証を行う Rust 製 CLI ツールです。

- エントリポイント: `generator/src/main.rs`
- 生成ロジック: `generator/src/generate.rs`
- 逆算法・延長法: `generator/src/backward.rs`
- 将棋ルール・詰み判定: `shogi-core/src/shogi.rs`
- df-pn ソルバー: `shogi-core/src/dfpn.rs`

## 1. 使い方

### サブコマンド

| コマンド | 説明 |
|----------|------|
| `generate` | パズル生成（デフォルト） |
| `validate` | 既存 JSON の検証 |
| `validate-fix` | 検証 + 不正パズルを自動削除 |

### 主なオプション（generate）

| オプション | 説明 |
|-----------|------|
| `--seed=N` | 乱数シード（省略時はシステム時刻） |
| `--only=N` | 特定の手数だけ生成（例: `--only=9`） |
| `--attempts9=N` | 9手詰の試行回数を指定 |
| `--extend-only` | 延長法のみ実行（ランダム法・逆算法をスキップ） |
| `--extend-from=N` | 延長元の手数を指定（例: `--extend-from=7`） |
| `--no-extend` | 延長法をスキップ |
| `--method=random\|backward\|both` | 生成方式の指定 |
| `--save-found-dir=DIR` | 中間結果を指定ディレクトリにも保存 |

### 実行例

```bash
# 全手数を生成
cd generator && cargo run --release -- generate --seed=12345

# 9手詰のみ、延長法で
cd generator && cargo run --release -- generate --only=9 --extend-only

# 検証
cd generator && cargo run --release -- validate
```

## 2. 生成パイプライン

`generate_puzzles()` は以下のフェーズを順に実行します。

### フェーズ0: 既存パズルの再検証

既存の `puzzles/N.json` を読み込み、現行ルールで再検証。不正なパズル（駒余り、手順長不一致等）は除外し、有効なものだけを保持します。

### フェーズ1: 逆算法（backward）

詰み局面から手を巻き戻して初期局面を構築します（`backward.rs`）。

1. ランダムに詰み局面を生成（`generate_mated_position`）
2. 攻め方の手を巻き戻す（`unwind_attacker_move`）
3. 守り方の手を巻き戻す（`unwind_defender_move`）
4. 2-3 を必要なペア数だけ繰り返す

巻き戻しのパターン:
- **攻め方**: 盤上の駒を元の位置に戻す、持ち駒に戻す、成りを戻す
- **守り方**: 玉の移動を戻す、王手駒の捕獲を戻す、合駒の挿入を戻す

### フェーズ2: 延長法（extend）

短手数のパズルを2手ずつ延長して長手数パズルを生成します。

- 7手詰パズルから1段延長 → 9手詰候補
- 9手詰パズルから1段延長 → 11手詰候補
- 3段以上の多段延長は成功率が極端に低いためスキップ

各ソースパズルから `extend_candidate()` で守り方1手+攻め方1手を巻き戻し、12回リトライして最良の候補を選択します。

手数に応じた試行数:
- 9手詰: ソースパズル1問あたり500試行
- 11手詰: ソースパズル1問あたり1000試行

### フェーズ3: ランダム法（random）

駒をランダムに配置し、詰将棋として成立するか検証します。

`random_candidate()` は守り方玉の近傍に攻め方・守り方の駒をランダム配置します。手数ごとのパラメータ:

| パラメータ | 1手詰 | 3手詰 | 5手詰 | 7手/9手 | 11手以上 |
|-----------|-------|-------|-------|---------|---------|
| 攻め方の駒数 | 1-2 | 2-3 | 2-4 | 3-5 | 4-6 |
| 守り方の駒数 | 0-1 | 0-2 | 1-3 | 1-3 | 1-4 |
| 持ち駒確率 | 20% | 30% | 40% | 50% | 50% |

### フェーズ4: 変異法（mutation）

フェーズ3で見つかったパズルの駒を微小変異させて新パズルを派生します。

変異操作（駒移動に3倍の重み）:
- **駒移動** (3×重み): ±1〜±2マス移動
- **駒種変更**: 別の駒種に置換
- **駒追加**: 守り方玉近傍に追加
- **駒削除**: 2枚以上ある場合のみ
- **持ち駒調整**: 攻め方持ち駒の1種を増減

5回に1回は左右反転（mirror）も試行します。

## 3. 検証・正規化

### validate_and_prune

候補局面は以下の順で検証・整形されます:

1. **quick_reject**: 安価な事前フィルタ（王手可能手数の上限、玉周囲の空きマス数等）
2. **攻め方玉の除去**: 詰将棋では攻め方玉は不要
3. **ソルバーで検証**: 5手以下は全幅探索、7手以上は df-pn
4. **手順長の確認**: ちょうど N 手の解であることを確認
5. **駒削減（prune）**: 駒を1つずつ除去し、成立する限り削る（7手以下のみ）
6. **未使用持ち駒の除去**: 解の手順で使われない攻め方持ち駒を削除
7. **駒余りチェック**: 最終局面で攻め方持ち駒が残っていたら不採用
8. **スライド駒の正規化**: スライド駒と玉の距離を約4マスに調整
9. **左右反転正規化**: 守り方玉が右上（x≤5）になるよう反転

### ソルバーの使い分け

| 手数 | ソルバー | 特徴 |
|------|---------|------|
| 1〜5手 | 全幅探索（`validate_tsume_puzzle`） | メモ化付きミニマックス、唯一解チェック内蔵 |
| 7手以上 | df-pn（`validate_tsume_dfpn_staged`） | 段階的バリデーション: 詰み判定→短手数不存在→唯一解チェック |

## 4. 重複・偏り抑制

`add_result()` で以下を同時チェック:

- **完全一致**: 局面 JSON の文字列比較
- **構造的一致**: 守り方玉からの相対配置 + スライド駒の距離を無視
- **駒構成上限**: 同一駒構成（攻め方+守り方+持ち駒）のパズルは最大3問
- **攻め方駒構成上限**: 攻め方の駒構成が同じパズルは最大3問

## 5. 多様性フィルタ（filter-puzzles.js）

生成済みパズルの中から、ユーザーに表出するパズルを選別するビルド時スクリプトです。

- `puzzles/*.json`（生データ、全問保持）→ `public/puzzles/*.json`（デプロイ用）
- 7手詰以上: 解の全手順が一致するパズルは1問に制限
- 同一駒構成のパズルは最大5問

延長法・変異法は同じ詰み筋のバリエーションを大量に生成するため、このフィルタが重要です。

## 6. 並べ替え（diversify_order）

生成済み候補は2種のスコアで評価され、易→難の順に並べられます:

- **score_puzzle**: 面白さ寄り（捨て駒、打ち、成り、持ち駒の多さ等）
- **difficulty_score**: 難しさ寄り（持ち駒量、大駒持ち、守備駒数など）

`diversify_order` で4分位に分割後、各分位内で特徴ベクトルの Manhattan 距離を使って似た問題の連続を減らします。

## 7. ハッシュベースのパズル ID

各パズルに初期局面から決定的に算出した8文字 hex のハッシュ ID を付与します。URL は `?mate=N&pid=HASH` 形式（`?id=N` も後方互換で動作）。

ハッシュ計算: 駒を正規化ソート → JSON 化 → djb2 ハッシュ → 下位32bit

## 8. 補足

- 乱数は独自 XORShift 系 `Rng` を利用（`shogi-core/src/rng.rs`）
- 並列化は `rayon::par_iter()` でバッチ単位に実施（バッチサイズ1000）
- 生成中は中間結果をバッチごとにファイルへ保存（途中 kill でも結果を失わない）
- Cargo workspace 構成: shogi-core（共有 lib）、wasm（cdylib）、generator（CLI）
