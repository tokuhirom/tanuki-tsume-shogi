# generator の実装解説

このディレクトリは、詰将棋パズルの自動生成と検証を行う Rust 製ツール (`tsume-gen`) です。

- エントリポイント: `generator/src/main.rs`
- 生成ロジック: `generator/src/generate.rs`
- 将棋ルール・詰み判定: `generator/src/shogi.rs`

## 1. 詰将棋ルール

本アプリが採用する詰将棋ルール:

1. **攻め方は王手の連続で詰ます** — 攻め方は毎手必ず王手をかけなければならない
2. **守り方は最善を尽くす** — 最長手数の応手（最長抵抗）を選ぶ
3. **守り方の持ち駒は駒箱から無制限に使える** — 盤上にも攻め方持ち駒にもない駒はすべて守り方が合駒として使用可能
4. **唯一解** — 攻め方の正解手が各局面で1通りのみ
5. **最短手数** — N手詰はN手でのみ詰み、N-2手以下では詰まない
6. **駒余りなし** — 最終局面で攻め方の持ち駒が残らない
7. **打ち歩詰め禁止** — 歩を打って詰みにする手は禁手
8. **二歩禁止** — 同じ筋に自分の歩が2枚存在してはならない
9. **行き場のない駒の禁止** — 打った駒が次に動けない位置への打ちは禁手

### 守り方の持ち駒の自動計算

将棋の全駒数（玉を除く）: 飛2, 角2, 金4, 銀4, 桂4, 香4, 歩18

```
守り方の持ち駒 = 全駒数 - 盤上の駒（成り駒は元の種類でカウント） - 攻め方の持ち駒
```

パズルJSONには守り方の持ち駒フィールドがあるが、ロード時に上記計算で上書きされる。

## 2. 使い方と実行フロー

`main.rs` は以下 2 サブコマンドを受けます。

- `generate`: パズル生成（デフォルト）
- `validate`: 既存 JSON の検証

`generate` の主なオプション:

- `--seed=`: 乱数シード
- `--attempts1=`, `--attempts3=`, `--attempts5=`, `--attempts7=`: 手数ごとの試行回数
- `--only=`: 特定の手数だけ生成する場合

実行時は `mate_len = [1, 3, 5, 7]` を順に処理し、結果を両方へ出力します。

- `puzzles/{mate_len}.json`
- `public/puzzles/{mate_len}.json`

### インクリメンタル生成

既存パズルを読み込み、新ルールで再検証して有効なものを保持する。不足分のみ新規生成する。

`validate` は `puzzles/*.json`（なければ `docs/puzzles/*.json`）を読み込み、全問について `validate_tsume_puzzle` で成立性を再確認し、駒余りチェックも実行する。

## 3. 生成パイプライン (`generate.rs`)

`generate_puzzles(seed, mate_length, attempts, curated_seeds, max, existing)` が中心。

1. 既存パズルの再検証・取り込み（invalidなものは除外）
2. 収集済み問題（curated）を取り込み
3. その左右反転も取り込み
4. ランダム生成 + 検証を rayon で並列実行
5. 見つかった局面から変異（mutation）を派生（各候補から20回）
6. 左右反転も試行
7. 変異フェーズ2（既存結果プールからランダム選択して変異, attempts/2 回）
8. 難易度順 + 多様性を考慮して並べ替え

### 3.1 候補局面の作り方

`random_candidate` は守り方玉の近傍に攻め方・守り方の駒をランダム配置する。手数ごとに `candidate_params` で以下を調整:

| パラメータ | 1手詰 | 3手詰 | 5手詰 | 7手詰 |
|-----------|-------|-------|-------|-------|
| 攻め方の駒数 | 1-2 | 2-3 | 2-4 | 3-5 |
| 守り方の駒数 | 0-1 | 0-2 | 1-3 | 1-4 |
| 持ち駒確率 | 低 | ~30% | ~40% | ~45% |

駒種は金・銀に偏重した重み付き配列から選択。

### 3.2 変異 (`mutate_initial`)

既存局面に対してランダム操作を 1 つ適用する（駒移動に3倍の重み）:

- **駒移動** (3×重み): ±1〜±2マス移動
- **駒種変更**: 別の駒種に置換
- **駒追加**: 守り方玉近傍に追加（10枚超で1枚削除）
- **駒削除**: 2枚以上ある場合のみ
- **持ち駒調整**: 攻め方持ち駒の1種を0か1に設定

### 3.3 検証・正規化 (`validate_and_prune`)

候補局面は次の順で整形:

1. 攻め方玉を除去（存在しても正規化で削る）
2. `validate_tsume_puzzle` で詰み成立を確認
3. `prune_initial` で不要駒を削減（成立する限り削る）
4. 解を再検証
5. 解再生後に未使用の攻め方持ち駒を初期局面から削除し再検証
6. **駒余りチェック**: 最終局面で攻め方持ち駒が残っていたら不採用
7. 守り方玉が左側なら左右反転して右側へ正規化
8. `score_puzzle` を計算

### 3.4 重複・偏り抑制

`add_result` で以下を同時チェック:

- 完全一致 (`sig_set`)
- 構造シグネチャ一致 (`struct_set`): 守り方玉からの相対配置 + 左右反転正規化
- 駒構成キーごとの出現上限（最大3）
- 攻め方駒構成キーごとの出現上限（最大3）

## 4. 並べ替え戦略（難易度 + 多様性）

生成済み候補には 2 種スコアがある:

- `score_puzzle`: 面白さ寄り（捨て駒、打ち、成り、着手先の多様さ等）
- `difficulty_score`: 難しさ寄り（持ち駒量、大駒持ち、守備駒数など）

最終順序は `diversify_order` で決定:

1. 難易度スコアで昇順
2. 4分位に分割
3. 各分位内で特徴ベクトル距離（Manhattan 距離）を使って greedy に散らす

全体として「易→難」の流れを保ちつつ、似た問題の連続を減らす。

## 5. 詰み判定ロジック (`shogi.rs`)

### 5.1 局面表現

- 盤: `State.board[81]`
- 持ち駒: `Hands { attacker[7], defender[7] }`
- 手番: `side_to_move`
- 玉位置キャッシュ: `attacker_king`, `defender_king`

`State::set` で玉キャッシュを同期する設計。

### 5.2 合法手生成

- `pseudo_moves_from`: 駒の擬似合法手（成り分岐込み）
- `pseudo_drops`: 打ち駒候補（行き場なし・二歩を除外）
- `legal_moves`: 自玉が王手になる手を除外、打ち歩詰め禁手も除外

打ち歩詰めは `pawn_drop_mate_forbidden` で、歩打ち後に相手へ合法手が無いかを確認して判定。

### 5.3 詰み探索 (`forced_mate_within`)

メモ化付きミニマックス探索:

- **攻め方手番**: 王手になる手のみ探索（詰将棋ルール）
- **守り方手番**: 全合法手を探索、1手でも逃れがあれば即不詰（early cutoff）
- **守り方の応手選択**: 最長抵抗（`>=` で最後の同長手を選ぶ）
- **メモ化**: FxHashMap, キー = hash(局面 + 残り手数)
- **結果**: `MateResult { mate, unique, line }`

### 5.4 問題としての成立条件

`validate_tsume_puzzle` は以下を満たすときのみ `Some(solution)` を返す:

- 手数が奇数
- 攻め方手番
- 守り方玉が存在
- 初期局面で守り方玉に王手がかかっていない
- 指定手数以内で詰む
- 2手短い手数では詰まない（最短性）
- 解が唯一

## 6. 探索アルゴリズムの改善余地

### 6.1 守り方の合駒生成の最適化（効果: 大）

現在 `pseudo_drops` は全81マス × 7駒種 = 567パターンをスキャンする。
守り方が合駒（interposition drop）する場合、有効な打ち場所は王手ラインの遮断マスのみ。

**改善案**: 王手がスライド駒（飛・角・香）による場合、遮断可能マスだけに限定して打ち駒を生成する。接触王手（隣接駒からの王手）には合駒不要なのでスキップ可能。

### 6.2 攻め方の手の並び替え（効果: 中）

現在は王手になる手を生成順のまま探索する。
捕獲手・成り手・玉に近い手を優先すれば、早い段階で詰み手順を見つけられ、枝刈りの機会が増える。

**改善案**: 王手の手をヒューリスティックでソート（捕獲 > 成り > 打ち > 移動）。

### 6.3 反復深化（効果: 中）

現在はいきなり最大深さまで探索する。
反復深化（1手→3手→5手→…）を使えば、浅い段階で不詰と判定できる局面を高速に除外できる。

**改善案**: `forced_mate_within` を反復深化で呼び出す。ただし候補生成時のみ適用（検証時は完全探索が必要）。

### 6.4 候補局面の品質向上（効果: 大 / 7手詰向け）

7手詰では合駒防御が強力で、ランダム局面がほぼ全て不詰になる。

**改善案**:
- **逆算生成**: 詰み局面から逆算して合法な初期局面を構築する
- **合駒封じの初期配置**: 守り方玉を端に配置して合駒マスを減らす（壁攻め）
- **スライド系の王手を減らす**: 金・桂など接触系の王手を優先（合駒で防げない）

### 6.5 守り方ドロップの遅延生成（効果: 中）

守り方の応手で、盤上の駒の移動だけで逃げられるならドロップの生成自体をスキップできる。
early cutoffと組み合わせれば、ドロップ生成コストを大幅に削減可能。

**改善案**: `legal_moves` で盤上の駒の手を先に生成し、逃れ手が見つかった時点でドロップ生成をスキップ。

### 6.6 メモ化テーブルの改善（効果: 小）

現在は1回の探索でメモを共有するが、クリアタイミングがない。
長時間生成ではメモが肥大化する。

**改善案**: パズル候補ごとにメモを新規作成（現状もそうなっている可能性あり。バッチ並列化の中で個別に作成されている）。

## 7. ハッシュベースのパズルID

各パズルに初期局面から決定的に算出した8文字hexのハッシュIDを付与。
URLは `?pid=HASH` 形式（`?id=N` も後方互換で動作）。

ハッシュ計算: 駒を正規化ソート → JSON化 → djb2ハッシュ → 下位32bit

## 8. curated データの読み込み

`load_curated("data/curated-puzzles.json")` は JSON の `"3"`, `"5"`, `"7"`, `"9"` キーを読み取り、`HashMap<u32, Vec<InitialData>>` として返す。

## 9. 補足

- 乱数は独自 XORShift 系 `Rng` を利用
- 並列化は `rayon::par_iter()` でバッチ単位に実施
- `validate` は全問を検証（NGでも中断せず最後まで実行）
- FxHashMap (rustc-hash v2) をメモ化テーブルに使用
