# 詰将棋生成アルゴリズムの調査と改善方針

## 1. 既知の生成手法

### 1.1 ランダム法（現行方式）

駒をランダムに配置し、詰将棋として成立するか検証する。成立したものだけを採用する。

**メリット**:
- 実装がシンプル
- 多様な局面を生成できる
- 変異（mutation）と組み合わせて効率化可能

**デメリット**:
- 長手数になるほどヒット率が極端に下がる
- 守り方の合駒を正しく考慮すると、さらにヒット率が低下
- 7手詰: 旧ルール（合駒なし）で20万試行→284問、新ルール（合駒あり）で20万試行→数問

### 1.2 逆算法（バックワード方式）

詰み局面から逆算して初期局面を構築する。

**手順（すぎゃーんメモの実装例）**:

1. **詰み局面の作成**
   - 後手玉を配置（1〜4筋、1〜4段に限定）
   - 周辺に先手駒を配置して王手状態を作る
   - 残りの38駒をランダム配置（二歩回避、行き場のない駒回避）
   - 約10回の試行で有効な詰み局面が得られる

2. **1手巻き戻し（1手詰の生成）**
   - 最終手の攻め方の駒を元の位置に戻す
   - 王手でない状態を復元する
   - 不要な駒を約8割削除
   - 成香・成桂などの慣習外の成駒を金等に置換

3. **3手詰以上への拡張**
   - 攻め方1手戻し → 守り方1手戻し → 攻め方1手戻し の繰り返し
   - 守り方の巻き戻しパターン: 玉の移動、王手駒の捕獲、合駒による遮断

**メリット**:
- 確実に詰む局面から出発するため効率が高い
- 長手数の問題を自然に生成できる
- 合駒で崩壊しにくい問題を作りやすい

**デメリット**:
- 実装が複雑（巻き戻しのパターンが多岐にわたる）
- 唯一解の保証が別途必要（Solver での検証が不可欠）
- 無駄合い判定が絡むと3手詰段階でも数秒〜数十秒の処理が必要

### 1.3 df-pn アルゴリズム（詰み判定の高速化）

proof number（証明数）と disproof number（反証数）を用いた探索手法。
詰将棋では証明数 = 玉の逃げ方の総数、反証数 = 攻め方の王手の総数。

**特徴**:
- 1525手詰の「ミクロコスモス」を解いた実績（脊尾昌宏氏の実装）
- 生成よりも「解く」側のアルゴリズムだが、検証フェーズの高速化に使える
- 現行のミニマックス + メモ化より効率的な場合がある

**参考実装**: KomoringHeights（ほぼ完璧な参考実装として知られる）

## 2. 現行アルゴリズムの問題分析

### 2.1 7手詰が生成できない原因

合駒ルール導入後、守り方が駒箱から自由に合駒を打てるため:

1. **分岐爆発**: 守り方の手に合駒7種 × 空きマス数が加わる
2. **詰み回避率の上昇**: スライド系王手のほぼ全てが合駒で防がれる
3. **ランダム局面の不適合**: ランダム配置では合駒防御しにくい局面が生まれにくい

### 2.2 無駄合い未実装の影響

- 守り方が手数を延ばすだけの合駒を打てるため、探索が深くなる
- 本来「詰み」の局面が「不詰」と判定されるケースはないが、手順が冗長化
- 最長抵抗の選択で無駄な合駒を含む長い手順が「最善」として選ばれる

## 3. 改善方針

### 3.1 短期: 無駄合い判定の簡易実装（効果: 大）

完全な無駄合い判定は複雑だが、簡易版で十分な効果が得られる:

**簡易判定**: 「合駒を同X（王手していた駒）で取った後、合駒しなかった場合と同じ手数以下で詰むなら無駄合い」

```
合駒が無駄合いか判定:
  1. 合駒の手を適用
  2. 王手していた駒で合駒を取る手を適用
  3. この局面から N-2 手以内で詰むか判定
  4. 合駒しなかった場合の局面からも N 手以内で詰むか判定
  5. 両方詰むなら無駄合い → 守り方の応手候補から除外
```

### 3.2 中期: 逆算法の導入（効果: 大 / 7手詰向け）

現行のランダム法と逆算法のハイブリッド:

1. 詰み局面をランダムに生成
2. 手を1つずつ巻き戻して初期局面を構築
3. 現行の `validate_tsume_puzzle` で唯一解・最短手数を検証
4. `prune_initial` で不要駒を削減

特に7手詰以上では逆算法が必須と考えられる。

### 3.3 中期: 候補局面の品質向上（効果: 中）

合駒が効きにくい局面を優先生成:

- **壁攻め**: 守り方の玉を端に配置（合駒マスが減る）
- **接触王手重視**: 金・銀・桂など隣接駒からの王手を優先（合駒不可）
- **スライド王手の制限**: 飛・角・香の王手を減らし、合駒の余地を狭める

### 3.4 長期: df-pn による検証高速化（効果: 中）

現行のメモ化ミニマックスを df-pn に置き換えることで:

- 長手数の詰み判定が高速化
- 不詰局面の早期棄却が改善
- 合駒を含む深い探索でも効率的に処理

## 4. 探索アルゴリズムの具体的改善案

### 4.1 合駒生成の最適化

現状: `pseudo_drops` が全81マス × 7駒種をスキャン

改善: スライド駒による王手の場合、遮断可能マス（玉と王手駒の間）だけに限定

```
飛車が横から王手 → 玉と飛車の間のマスにのみ合駒可能
角が斜めから王手 → 玉と角の間のマスにのみ合駒可能
接触王手（隣接駒） → 合駒不可、スキップ
```

### 4.2 攻め方の手の並び替え

現状: 王手になる手を生成順のまま探索

改善: ヒューリスティックでソート（早期枝刈りを促進）
- 捕獲手（駒を取る王手）を最優先
- 成り王手を次に
- 打ち王手（合駒で防がれやすい）を後回し

### 4.3 守り方ドロップの遅延生成

現状: `legal_moves` で盤上の駒の移動とドロップを一括生成

改善: 盤上の駒の手を先に生成し、逃れ手が見つかった時点で即座に返す（ドロップ生成をスキップ）

### 4.4 反復深化

現状: いきなり最大深さまで探索

改善: 1手→3手→5手→7手と段階的に深化（浅い段階で不詰と判明すれば打ち切り）
